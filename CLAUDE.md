## Project Overview

MemoryLane is a system tray Electron application that captures screenshots at regular intervals. Built with TypeScript using electron-vite for development and electron-builder for packaging. The application runs as a tray-only app (no main window) and captures screenshots based on user interaction and visual changes. Summaries are generated by an LLM (OpenRouter by default, or custom OpenAI-compatible endpoint) and stored locally in SQLite.

## Architecture

### Dual Entry Points

The app has two separate entry points, both built by electron-vite:

1. **`src/main/index.ts`** — Electron tray app (main process). Creates system tray, manages capture lifecycle, renders settings UI.
2. **`src/main/mcp-entry.ts`** — Standalone MCP server. Runs under `ELECTRON_RUN_AS_NODE=1` as a plain Node process so it doesn't register as a second Electron app instance. MCP clients launch the Electron binary with this script to get stdio-based MCP transport.

Both use the same Electron binary to keep native module ABIs compatible.

### Core Pipeline Flow

```
User interaction → InteractionMonitor → EventCapturer (event windows)
    → PipelineHarness (screenshot capture per window)
    → ActivityExtractor (groups frames into activities)
    → ActivityTransformer (OCR + semantic summarization via LLM)
    → SqliteActivitySink (persists to SQLite)
    → ActivityCleanup (deletes processed media)
```

Key pipeline modules:
- **`event-capturer.ts`** — Groups user interactions into event windows (closed by idle gap, app change, or max duration)
- **`pipeline-harness.ts`** — Orchestrates the capture-to-storage pipeline; coordinates screen capturer, event windowing, and activity processing
- **`activity-extractor.ts`** — Extracts activities from event windows with associated screenshots
- **`activity-transformer.ts`** — Runs OCR and semantic LLM summarization on activities
- **`activity-semantic-service.ts`** — LLM client supporting video and image modes, OpenRouter and custom endpoints, with fallback chains

### Composition Root

**`src/main/runtime.ts`** (`createMainRuntime()`) wires together all services: storage, capture controller, OCR, embeddings, semantic service, API key management, and usage tracking. Both the tray app entry point and tests use this factory.

### Key Directories

```
src/main/
├── recorder/         # Screen capture, app-watcher, interaction monitoring
├── processor/        # OCR (macOS Vision + Windows native), embeddings (Transformers.js)
├── storage/          # SQLite (better-sqlite3 + sqlite-vec), migrations, repositories
├── semantic/         # LLM client, prompt construction, model chains, debug dumps
├── video/            # FFmpeg-based video stitching for video-mode pipeline
├── settings/         # API keys, capture settings, custom endpoints, device identity
├── services/         # Managed key provisioning, usage tracking, pattern detection, DB export
├── integrations/     # Claude Desktop, Claude Code, Cursor, Slack
├── ui/               # Tray menu, main window, permissions dialogs
├── mcp/              # MCP server (tools: search_context, browse_timeline, get_event_details)
src/renderer/         # React UI for settings window (Tailwind v4 + shadcn/ui)
src/shared/           # Shared types and constants
packages/cli/         # Standalone CLI package (@deusxmachina-dev/memorylane-cli)
native/windows/       # Rust sidecar binaries (app-watcher, screenshot-capturer)
```

### Native Sidecars (`native/`)

First-party native binaries (separate from native Node.js addons in `node_modules`):

- **`native/windows/app-watcher/`** — Rust. Monitors foreground window changes and browser URLs on Windows.
- **`native/windows/screenshot-capturer/`** — Rust. Native screenshot capture daemon for Windows.
- macOS equivalents use Swift (built via `build:swift`); Windows uses Rust (built via `build:rust`).

### Native Node.js Modules

These require special handling (externalized in Vite, rebuilt for Electron via `postinstall`, unpacked from ASAR):

- `uiohook-napi` — Keyboard/mouse monitoring
- `sharp` — Image processing
- `better-sqlite3` — SQLite database
- `onnxruntime-node` — ML inference
- `sqlite-vec` — Vector search extension

### Single-Runtime Environment

All scripts run under Electron's Node.js via `scripts/enode.js` (Windows) or `scripts/enode.sh` (macOS/Linux). These wrappers set `ELECTRON_RUN_AS_NODE=1` and execute through Electron's bundled Node.js binary. Native modules are compiled once for Electron during `npm install` (via postinstall) and never need rebuilding.

### Build System

- **`electron.vite.config.ts`** — Unified Vite config for main, preload, and renderer. Builds both entry points (`index.ts` and `mcp-entry.ts`).
- **`electron-builder.yml`** — Packaging config for macOS, Windows, Linux.
- **`vitest.config.ts`** — Test config. Tests run in node environment with `better-sqlite3` externalized.

### Path Aliases

Configured in both Vite and TypeScript:
- `@/` → `src/`
- `@components/` → `src/renderer/components/`
- `@constants` → `src/shared/constants`
- `@types` → `src/shared/types`
- `@assets` → `assets/` (renderer only)

Use these aliases instead of deep relative paths in renderer code.

### Storage

SQLite database (`memorylane.db` / `memorylane-dev.db`) in Electron's `userData` directory. Uses:
- `better-sqlite3` for persistence and full-text search (FTS5 with sync triggers)
- `sqlite-vec` for vector similarity search over embeddings
- Numbered migration files in `src/main/storage/migrations/`

### Renderer

React 19 + Tailwind v4 + shadcn/ui components. Single window (`main-window.html`) for settings/status. The renderer communicates with main process via preload-exposed `MainWindowAPI`.

## Key Patterns

### Tray App Behavior

- No main window created by default (pure tray app)
- Dock icon hidden on macOS
- App doesn't quit when all windows close
- Tray menu dynamically updates based on capture state

### Screenshot Data Structure

Each captured screenshot provides:
- `id`: UUID
- `filepath`: Absolute path to PNG file
- `timestamp`: Unix timestamp in milliseconds
- `display`: Display metadata (id, width, height)
- `trigger`: Capture reason (manual, baseline_change, etc.)

### Test Conventions

- Co-located tests: `module.test.ts` next to `module.ts`
- Integration tests: `module.integration.test.ts`, gated by env vars
- Storage tests use `test-utils.ts` for in-memory SQLite setup
